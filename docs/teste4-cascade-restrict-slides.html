CASCADE e RESTRICT

Regras de integridade referencial no Sequelize

Pressione → ou Space para começar

❓ O Problema

Quando temos associações entre modelos, surge a pergunta:

O que acontece quando excluímos um registro que tem registros filhos?
🔄 CASCADE
A exclusão do pai é propagada para os filhos.
Filhos são excluídos junto.
🚫 RESTRICT
A exclusão do pai é bloqueada se existirem filhos.
Lança um erro.

🖼️ CASCADE vs RESTRICT — Visualmente

⚙️ Configurando as Regras

JavaScript 
// Cliente → Telefone: CASCADE
// Excluir cliente → exclui todos os telefones
Cliente.hasMany(Telefone, {
    as: 'telefones',
    onDelete: 'CASCADE',
    onUpdate: 'CASCADE'
});

// Venda → Cliente: RESTRICT
// Excluir cliente com vendas → ERRO!
Venda.belongsTo(Cliente, {
    onDelete: 'RESTRICT',
    onUpdate: 'RESTRICT'
});
As opções onDelete e onUpdate definem o comportamento quando o registro pai é excluído ou alterado.

🗺️ Diagrama com Regras

Mesmo cliente, comportamentos diferentes em cada associação ao ser excluído.

💥 O que Acontece ao Excluir?

JavaScript 
// Tentando excluir o cliente que tem telefones E vendas
await cliente1.destroy();
Telefones (CASCADE) ✅
Os 3 telefones seriam excluídos automaticamente.
Vendas (RESTRICT) ❌
Existem vendas → exclusão bloqueada!
Erro 
SequelizeForeignKeyConstraintError
→ Não é possível excluir o cliente: existem vendas com RESTRICT
→ O RESTRICT bloqueia a operação para proteger dados financeiros
O RESTRICT prevalece! O cliente não pode ser excluído enquanto houver vendas vinculadas.

🔄 Se Só Houvesse CASCADE…

Se não houvesse vendas (ou se vendas também fosse CASCADE), ao excluir o cliente:

CASCADE = "efeito dominó" — a exclusão do pai propaga em cascata para todos os filhos.

🛡️ RESTRICT em Ação

Com RESTRICT nas vendas, ao tentar excluir o cliente:

RESTRICT = "escudo protetor" — impede a exclusão para proteger os dados importantes (vendas, transações, históricos).

🔧 Outras Opções Disponíveis

Exemplos 
// SET NULL — Define FK como null ao excluir pai
Cliente.hasMany(Telefone, { onDelete: 'SET NULL' });

// SET DEFAULT — Define FK como valor padrão
Cliente.hasMany(Telefone, { onDelete: 'SET DEFAULT' });

// NO ACTION — Similar ao RESTRICT (depende do banco)
Cliente.hasMany(Telefone, { onDelete: 'NO ACTION' });
✅

Conclusão

Nesta aula, aprendemos sobre regras de integridade referencial:

  • CASCADE — propaga exclusões/atualizações do pai para os filhos
  • RESTRICT — bloqueia exclusões quando existem filhos
  • As regras são definidas com onDelete e onUpdate
  • O RESTRICT prevalece sobre o CASCADE quando conflitam
  • Use CASCADE para dados dependentes (telefones, configs)
  • Use RESTRICT para dados importantes (vendas, logs)
Escolher a regra correta é fundamental para a integridade e consistência dos dados.
