Introdução

Quando temos associações entre modelos, surge a questão: o que acontece quando tentamos excluir ou atualizar um registro que tem registros dependentes?

O Sequelize oferece duas estratégias principais para tratar essa situação:

  • CASCADE: A exclusão/atualização do registro pai é propagada para os registros filhos
  • RESTRICT: A exclusão/atualização do registro pai é bloqueada se existirem registros filhos

Definindo os Modelos

const { Sequelize, Model, DataTypes } = require("sequelize");

const sequelize = new Sequelize({
    dialect: 'sqlite',
    storage: 'database.sqlite'
});

// Definindo as classes de modelo
class Cliente extends Model {
    static init(sequelize) {
        super.init({
            nome: DataTypes.STRING
        }, { sequelize, modelName: 'cliente', tableName: 'clientes' })
    }
}

class Telefone extends Model {
    static init(sequelize) {
        super.init({
            numero: DataTypes.STRING
        }, { sequelize, modelName: 'telefone', tableName: 'telefones' })
    }
}

class Venda extends Model {
    static init(sequelize) {
        super.init({
            valor: DataTypes.FLOAT
        }, { sequelize, modelName: 'venda', tableName: 'vendas' })
    }
}

// Inicializando os modelos
Cliente.init(sequelize);
Telefone.init(sequelize);
Venda.init(sequelize);

Os três modelos são idênticos ao tutorial de associações. A diferença estará nas configurações das associações, onde definiremos as regras de CASCADE e RESTRICT.

Configurando Associações com Regras de Integridade

// Configurando as associações entre os modelos
Cliente.hasMany(Telefone, {as: 'telefones', onDelete: 'CASCADE', onUpdate: 'CASCADE'});
Venda.belongsTo(Cliente, {onDelete: 'RESTRICT', onUpdate: 'RESTRICT'});

Neste trecho, definimos as mesmas associações do teste anterior, mas com regras adicionais:

  1. Cliente → Telefone (hasMany com CASCADE):
    • onDelete: 'CASCADE': Ao excluir um cliente, seus telefones são excluídos automaticamente
    • onUpdate: 'CASCADE': Ao alterar o ID de um cliente, a FK nos telefones é atualizada automaticamente
  2. Venda → Cliente (belongsTo com RESTRICT):
    • onDelete: 'RESTRICT': Impede a exclusão de um cliente que tenha vendas vinculadas
    • onUpdate: 'RESTRICT': Impede a alteração do ID de um cliente que tenha vendas vinculadas
🔄 CASCADE (Cliente → Telefone):

Excluir o cliente? → Todos os telefones dele são excluídos junto!

Útil quando os filhos não fazem sentido sem o pai.

🚫 RESTRICT (Venda → Cliente):

Excluir o cliente? → ERRO! Existem vendas vinculadas a ele.

Útil quando os filhos devem ser preservados (registros financeiros, históricos, etc.).

Inserindo Dados de Exemplo

Inserindo Cliente e Telefones

// Inserindo um cliente e dois telefones para, depois, vincular os telefones ao cliente
const cliente1 = await Cliente.create({ nome: "Alberto" });
const telefone1 = await Telefone.create({ numero: "(11) 1111-1111" });
const telefone2 = await Telefone.create({ numero: "(22) 2222-2222" });
await cliente1.setTelefones([telefone1, telefone2]);

// Inserindo um telefone já com informação do cliente
const telefone3 = await Telefone.create({ numero: "(33) 3333-3333", clienteId: cliente1.id });

Criamos um cliente e três telefones vinculados a ele. Idêntico ao tutorial de associações.

Inserindo Vendas

// Inserindo duas vendas para, depois, vincular as vendas ao cliente
const venda1 = await Venda.create({ valor: 100 });
const venda2 = await Venda.create({ valor: 200 });
await venda1.setCliente(cliente1);
await venda2.setCliente(cliente1);

// Inserindo uma venda já com informação do cliente
const venda3 = await Venda.create({ valor: 300, clienteId: cliente1.id });

Criamos três vendas vinculadas ao mesmo cliente Alberto.

Testando a Exclusão: O que Acontece?

// Tentando excluir o cliente
await cliente1.destroy();

Ao tentar excluir o cliente Alberto, o Sequelize precisa verificar as regras definidas nas associações:

  1. Telefones (CASCADE): Os 3 telefones do cliente seriam excluídos automaticamente ✅
  2. Vendas (RESTRICT): Como existem vendas vinculadas, a exclusão é bloqueada com erro ❌

Resultado: O RESTRICT prevalece! O cliente não pode ser excluído porque existem vendas vinculadas a ele.

❌ SequelizeForeignKeyConstraintError
→ Não é possível excluir o cliente porque existem vendas com RESTRICT
→ O RESTRICT bloqueia a operação para proteger os dados financeiros

O Comportamento CASCADE em Ação

Se NÃO houvesse vendas (ou se vendas também fosse CASCADE), ao excluir o cliente:

  • O cliente Alberto seria excluído
  • (11) 1111-1111 seria excluído automaticamente
  • (22) 2222-2222 seria excluído automaticamente
  • (33) 3333-3333 seria excluído automaticamente

O CASCADE propaga a exclusão em cascata para todos os registros filhos.

O Comportamento RESTRICT em Ação

Como existem vendas com RESTRICT, ao tentar excluir o cliente:

  • A operação é bloqueada
  • O cliente NÃO é excluído
  • Os telefones NÃO são excluídos (a cascata nem inicia)
  • As vendas continuam intocadas
  • Um erro SequelizeForeignKeyConstraintError é lançado

O RESTRICT protege a integridade dos dados impedindo exclusões perigosas.

Comparação: CASCADE vs RESTRICT

Característica CASCADE RESTRICT
Comportamento ao excluir pai Exclui filhos automaticamente Bloqueia a exclusão (erro)
Comportamento ao atualizar pai Atualiza FK dos filhos Bloqueia a atualização (erro)
Protege dados filhos? Não — filhos são perdidos Sim — filhos são preservados
Uso comum Telefones, endereços, configurações Vendas, transações, históricos
SQL gerado ON DELETE CASCADE ON DELETE RESTRICT
Quando usar Filhos não fazem sentido sem o pai Filhos devem ser preservados mesmo sem o pai

Outras opções disponíveis no Sequelize:

  • SET NULL: Define a FK do filho como NULL ao excluir o pai
  • SET DEFAULT: Define a FK do filho como o valor padrão ao excluir o pai
  • NO ACTION: Similar ao RESTRICT (depende do banco de dados)

Conclusão

Neste tutorial, aprendemos sobre as regras de integridade referencial no Sequelize:

  • CASCADE propaga exclusões/atualizações do pai para os filhos
  • RESTRICT bloqueia exclusões/atualizações quando existem registros filhos
  • As regras são definidas nas opções onDelete e onUpdate das associações
  • O RESTRICT é útil para proteger dados importantes (vendas, transações, logs)
  • O CASCADE é útil para dados dependentes (telefones, endereços, configurações)

Escolher a regra correta é fundamental para manter a integridade e a consistência dos dados no seu banco de dados.
